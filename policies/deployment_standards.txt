"""Deployment Standards Document

Standards for reliable, safe, and efficient deployments to production.

VERSION: 1.0
APPLICABLE TO: All application and service deployments
"""

DEPLOYMENT STRATEGIES

1. BLUE-GREEN DEPLOYMENT
   Requirements:
   - Maintain two identical production environments
   - Deploy to inactive environment (green)
   - Run full test suite against green
   - Switch traffic to green in single operation
   - Keep blue as instant rollback option
   - Monitor green for 24 hours before decommissioning

2. CANARY DEPLOYMENT
   Requirements:
   - Deploy to 5% of traffic first
   - Monitor metrics: error rate, latency, resource usage
   - Increase percentage at 15-minute intervals if healthy
   - Rollback threshold: >0.5% error rate increase
   - Keep release notes accessible
   - Rollback capability available throughout

3. ROLLING DEPLOYMENT
   Requirements:
   - Deploy to subset of instances sequentially
   - Health checks pass on each instance before continuing
   - Load balancer removes unhealthy instances
   - Maintain 50%+ of capacity during deployment
   - Maximum 10-minute window between instances
   - Rollback if any instance fails health check

4. FEATURE FLAGS (RECOMMENDED)
   Requirements:
   - Code deployed, features hidden behind flags
   - Gradual enablement by user segment
   - Monitoring flag usage and impact
   - Disable within 1 minute if needed
   - Clean up flags after 30 days in production


MONITORING REQUIREMENTS

1. PRE-DEPLOYMENT MONITORING
   Required metrics:
   - Baseline error rate, latency, throughput
   - Resource utilization (CPU, memory, disk)
   - Active user count trending
   - Dependency health checks

2. DEPLOYMENT MONITORING (CRITICAL)
   Requirements:
   - Dashboard visible during deployment
   - Real-time metrics streaming
   - Alerting configured for anomalies
   - SLA monitoring active
   - Rollback capability confirmed ready
   
   Monitored metrics:
   - Error rate (alert at 2x baseline)
   - P99 latency (alert if +30%)
   - CPU/Memory usage
   - Database connection pool
   - Message queue depth
   - Cache hit ratio

3. POST-DEPLOYMENT MONITORING (24+ HOURS)
   Requirements:
   - Metrics stable and healthy
   - No increase in support tickets
   - No performance degradation
   - No security issues detected
   - Customer feedback review


COMMUNICATION AND COORDINATION

1. DEPLOYMENT SCHEDULE ANNOUNCEMENT
   Timeline:
   - Announce deployment 1 week before (if major change)
   - Send reminder 48 hours before
   - Daily notification for the day of
   - During deployment: hourly status
   - Post-deployment: success notification
   
   Channels:
   - Team Slack channels
   - Email to stakeholders
   - On-call team via PagerDuty
   - Customer-facing status page

2. COORDINATION WITH TEAMS
   Required before deployment:
   - Customer support notified (escalation procedures)
   - Product team prepared for customer inquiries
   - Infrastructure team ready for rollback
   - Database team standing by (if DB changes)
   - Security team available (if security changes)

3. INCIDENT COMMUNICATION
   If issues detected:
   - Status page updated immediately
   - All stakeholders notified within 5 minutes
   - Root cause investigation started
   - Hourly updates provided
   - Retrospective within 24 hours


DATABASE DEPLOYMENT

1. SCHEMA CHANGES
   Requirements:
   - Backward compatible schema (must support old application simultaneously)
   - Tested with production data volume
   - Rollback verified (include reverse migration script)
   - Deadlock conditions checked
   - Long-running queries monitored
   - Maintenance window if necessary (notify users)

2. DATA MIGRATIONS
   Requirements:
   - Migration script idempotent (can run multiple times safely)
   - Estimate runtime with production data
   - Parallel processing for large tables
   - Rename tables atomically
   - Constraint validation after migration
   - Backups taken at each step
   - Data integrity verification queries

3. DEPENDENCY UPDATES
   Requirements:
   - Driver/library version compatibility verified
   - Connection pool settings optimized
   - Query timeouts configured
   - Slow query logs analyzed
   - Connection limits adjusted if needed


HEALTH CHECK STANDARDS

1. STARTUP CHECKS
   Requirements:
   - Application starts without errors
   - Configuration loads successfully
   - Database connections established
   - Cache connections ready
   - External service connectivity tested
   - Timeout: 60 seconds maximum

2. READINESS CHECKS
   Requirements:
   - All dependencies available
   - Configuration valid
   - Database responsive
   - Cache responsive
   - Service ready to accept traffic
   - Performed every 10 seconds

3. LIVENESS CHECKS
   Requirements:
   - Main process running
   - No deadlocks detected
   - Memory usage reasonable
   - Response time acceptable
   - Performed every 5 seconds
   - Failure triggers restart


CAPACITY PLANNING

1. RESOURCE REQUIREMENTS
   Requirements:
   - Peak load capacity estimated
   - Resource limits configured (CPU, memory)
   - Autoscaling thresholds set appropriately
   - Reserve capacity of 30% maintained
   - Growth projections reviewed quarterly

2. LOAD TESTING
   Requirements:
   - Simulate 150% of expected peak load
   - Test failure scenarios (dependency failures)
   - Test scale-down procedures
   - Test database query performance under load
   - Test API rate limiting
   - Document results and pass/fail criteria


VERSION MANAGEMENT

1. VERSION TAGGING
   Requirements:
   - Semantic versioning: MAJOR.MINOR.PATCH
   - Git tags for all releases
   - Release notes document changes
   - Breaking changes clearly marked
   - Dependency versions documented

2. COMPATIBILITY
   Requirements:
   - API backward compatibility maintained
   - Database compatibility maintained
   - Client library compatibility maintained
   - Graceful deprecation period (minimum 2 versions)

3. ROLLBACK VERSIONS
   Requirements:
   - Previous 3 versions kept available
   - Version comparison docs maintained
   - Rollback tested (not just assumed)
   - Dependency versions match previous deployment


ROLLBACK AND RECOVERY

1. AUTOMATED ROLLBACK TRIGGERS
   Requirements:
   - Error rate >2x baseline triggers rollback
   - P99 latency >30% increase triggers rollback
   - Health check failures trigger restart
   - Unhealthy instances removed from load balancer

2. MANUAL ROLLBACK PROCEDURE
   Requirements:
   - Single command/click to initiate
   - Completes within SLA (target <5 minutes)
   - Data validation after rollback
   - Communication sent to stakeholders
   - Root cause investigation logged

3. DISASTER RECOVERY
   Requirements:
   - Backup restore tested monthly
   - RTO (Recovery Time Objective): <1 hour
   - RPO (Recovery Point Objective): <15 minutes
   - Database consistency verified post-recovery
   - Infrastructure rebuilt in <2 hours

- [ ] Security review passed (if applicable)
- [ ] Documentation updated
- [ ] Rollback procedure tested

- [ ] Feature flags configured (if using)
- [ ] Deployment window acceptable
- [ ] No ongoing incidents
- [ ] All systems healthy (deployments dependencies)
